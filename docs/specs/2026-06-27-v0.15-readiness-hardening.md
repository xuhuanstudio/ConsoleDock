# v0.15 Readiness Hardening

`v0.15` is a focused hardening release for the existing local debug SDK. It improves privacy defaults, lifecycle consistency, validation coverage, and documentation accuracy without expanding ConsoleDock into analytics, telemetry, remote upload, crash reporting, network inspection, or a test automation platform.

## Goals

- Protect oversized partial logs when sensitive keys or values are split across fragments.
- Apply baseline obvious-secret redaction to local report-adjacent text that does not pass through the core log store.
- Keep local Debug Action execution history bounded and session-scoped.
- Keep Objective-C/UIKit startup behavior aligned with the Swift facade.
- Ensure public API validators and versioned docs track the actual exposed API.
- Document remaining 1.0 readiness items without forcing broad API redesign into this release.

## Changes

- Core partial-fragment redaction now carries per-source continuation state and a short tail so split sensitive keys such as `Authorization` and values after `token=` are handled conservatively.
- App Context values, Debug Action summaries/messages, and Local Session Archive notes receive baseline obvious-secret redaction before they appear in ConsoleDock reports.
- Debug Action execution history is bounded to the newest 500 records, and unregistering an action clears its session-only recent parameter values.
- UIKit copy actions use local-only expiring pasteboard writes on supported iOS versions.
- ConsoleDock-owned temporary report files and archive files apply file protection where available and are excluded from backup.
- Objective-C/UIKit `start` delegates Swift facade side effects so session state resets consistently.
- Swift and Objective-C API surface validators include floating button position diagnostics.
- Versioned README validation derives the current release tag from `CHANGELOG.md` and checks stable guidance snippets instead of brittle release paragraphs.

## Out Of Scope

- Analytics, telemetry, user statistics, or automatic upload.
- Background or continuous raw log file persistence.
- Crash reporting, network capture, remote commands, or automatic routing.
- CocoaPods, XCFramework, or new distribution channels.
- Marker data-model redesign.
- Public API freezing for `1.0`.

## 1.0 Follow-Up

- Audit public initializer visibility and long-term model stability.
- Revisit `Configuration` equality semantics because redactor closures cannot be meaningfully compared.
- Review whether marker entries need a first-class internal kind instead of relying on the `[marker]` message prefix.
- Decide whether low-level `ConsoleDockCore` public exposure should remain exactly as-is for stable `1.0`.
