# ConsoleDock v0.12.0 Public Readiness And Visual QA

## Summary

`v0.12.0` is a public-readiness and visual-quality release. It should not add another major debugging subsystem. The project already has the core local workflow: Logs, Debug Actions, parameter forms, App Context, Issue Reports, Session Timeline, and Local Session Archive. The next real gap is that the public documentation and screenshot assets still present an early Logs-only product.

This release should make the repository look and validate like the product it has become:

- update public screenshots from real iOS Simulator runs;
- document the current local debugging workflow visually and textually;
- add lightweight validation so stale or missing visual assets are caught before release;
- keep simulator UI smoke tests as behavioral validation while adding an explicit visual QA path;
- do a focused 1.0-readiness pass on public docs, release process, and API surface wording.

## Problem

The current README screenshot, `docs/assets/swift-sample-console.png`, was last changed near the first public preview. It shows an early Logs list and does not communicate the current shipped surface:

- Logs with structured query and jump controls;
- Actions with registered local shortcuts and parameter forms;
- Timeline for markers, actions, and error/fault entries;
- Context tab for app-provided diagnostics;
- Issue Report and Local Session Archive flows.

Automated simulator UI smoke tests still run locally and in GitHub release validation, but they only prove targeted flows are operable through accessibility identifiers. They do not prove that public screenshots are current, visually coherent, or useful to a first-time reader.

## Goals

- Replace the stale README hero screenshot with current Simulator output.
- Add a small curated screenshot set that shows the current product shape without turning docs into a gallery.
- Create a repeatable local screenshot capture workflow for the Swift sample app.
- Add a validator that ensures expected screenshot assets exist, are tracked intentionally, have reasonable dimensions, and are referenced by docs.
- Update release-process docs so visual QA is an explicit pre-release task when UI changes materially.
- Review README, simplified Chinese README, DocC entry points, sample walkthrough, roadmap, and changelog for stale visual or version wording.
- Keep `v0.12.0` focused on readiness and trust, not new debugging behavior.

## Non-Goals

- No redesign of the UIKit console.
- No SwiftUI rewrite.
- No new debugging panel or new product mode.
- No network inspector, crash reporter, remote upload, remote commands, route discovery, CocoaPods, or XCFramework work.
- No promise of complete Swift `Logger`, `os_log`, or Apple unified logging capture.
- No generated screenshots from private apps or production data.
- No large visual brand system or marketing site.

## Screenshot Capture Workflow

Add a focused script, for example:

```sh
scripts/capture-swift-sample-screenshots.sh
```

The script should:

- pick the same iPhone Simulator selection strategy used by the UI smoke scripts, while allowing `CONSOLEDOCK_UI_SMOKE_DESTINATION`;
- launch the Swift sample in deterministic UI automation mode;
- drive only the minimal flow needed to open ConsoleDock and visit Logs, Actions, Timeline, and Archive surfaces;
- save PNG files under `docs/assets/`;
- avoid secrets, user-specific paths, production data, and environment-specific names inside the screenshots;
- fail clearly if `xcodebuild`, Simulator boot, UI automation, or screenshot capture fails.

Implementation can either extend the existing Swift sample UI test target with screenshot-specific test methods or use a separate script plus `xcrun simctl io booted screenshot` at stable UI states. Prefer reusing XCTest where possible because existing accessibility identifiers already cover the needed screens.

## Visual QA Validation

Add a lightweight repository validator, for example:

```sh
python3 scripts/validate-doc-assets.py
```

It should verify:

- required screenshot files exist;
- screenshot files are tracked intentionally and allow-listed by release content audit;
- each required image is a PNG with non-trivial dimensions;
- README and sample walkthrough reference the current screenshot set;
- stale references to removed screenshot names are not present;
- assets do not live under generated build paths.

The validator should be part of `scripts/validate-release.sh`.

This validator should not attempt subjective image analysis. It should catch stale/missing asset hygiene issues, while manual visual QA remains responsible for judging whether screenshots are useful and current.

## Manual Visual QA

Update `docs/release-process.md` so minor releases that materially affect UIKit UI require a visual QA pass:

1. Run the screenshot capture script on a local machine with an available iOS Simulator.
2. Open the resulting screenshots.
3. Confirm they show the current shipped surfaces and redacted deterministic sample content.
4. Confirm no private data, local filesystem paths, or production logs are visible.
5. Confirm README and sample walkthrough still render correctly.

This is intentionally separate from UI smoke. UI smoke proves behavior; visual QA proves public presentation.

## API And Compatibility Review

`v0.12.0` should include a focused public API review, but only documentation or validator changes should be made unless a real issue is found.

Review:

- Swift facade naming for debug actions, app context, issue reports, timeline history, and session archives;
- Objective-C/UIKit facade selectors and nullability wording;
- release-build safety wording;
- local-only persistence wording for archives;
- `v0.x` preview language versus stable `1.0` promises.

Do not introduce breaking API changes unless the review finds a clear pre-1.0 correction that is worth doing before stabilization.

## Release Criteria

`v0.12.0` can ship when:

- current screenshots are captured from the Swift sample app and committed;
- README and sample walkthrough use the current screenshots;
- release content audit intentionally allow-lists only the approved screenshot assets;
- visual asset validation runs in release validation;
- simulator UI smoke still passes for Swift and Objective-C samples;
- public docs no longer make the project look Logs-only;
- no new unbounded persistence, remote behavior, or distribution claims are introduced.

## Why Not 1.0 Yet

The product capability is close to a stable SDK, but the public presentation and visual validation loop are not yet at 1.0 quality. `v0.12.0` should close that gap without changing the product shape. If this release lands cleanly and no API issues are found during the readiness review, the next milestone can reasonably be `v1.0.0`.
