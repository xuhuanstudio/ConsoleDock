# ConsoleDock v0.6 Daily Debug Usability

Status: accepted design specification for the v0.6.0 implementation phase.

## Summary

ConsoleDock v0.6.0 should improve the parts of the product that testers use repeatedly during a local mobile debug session. The project already has capture, explicit logging, log detail, local issue reports, Debug Actions, Swift and Objective-C samples, and release validation. The remaining daily-use friction is not another distribution channel or a larger platform. It is opening the console in the right way, finding important entries quickly, and navigating registered actions when an app has more than a few of them.

v0.6.0 focuses on three bounded improvements:

1. Configurable and controllable floating console trigger.
2. Log list jump affordances for latest and first visible error/fault entries.
3. Search in the Debug Actions page.

## Goals

- Let apps choose where the floating `CD` button starts so it does not cover common test controls.
- Let apps hide or show the bundled floating trigger at runtime without stopping ConsoleDock.
- Keep `ConsoleDock.showConsole()` useful for apps that want their own debug entry point and do not want the bundled floating button visible.
- Add a small log jump menu so testers can return to the latest visible log or first visible error/fault after filtering.
- Add local Actions search so larger debug menus stay usable without adding parameters, remote commands, persistence, or automation-platform behavior.
- Keep the UIKit implementation source-first, local-only, debug/test-first, and SPM-only.

## Non-Goals

- Shake-to-open in v0.6.0. A reliable universal shake trigger can require key-window or responder-chain intervention, which is too invasive for this release.
- Default persistence of button position or user filters.
- Remote commands or remotely delivered Debug Actions.
- Debug Action parameters, async state management, routing discovery, or automatic page navigation.
- Network inspection, crash reporting, CocoaPods, XCFrameworks, or SwiftUI rewrite.
- Complete Swift `Logger`, `os_log`, or Apple unified logging capture.

## Public API

### Floating Button Position

Swift:

```swift
let configuration = ConsoleDock.Configuration(
    floatingButtonPosition: .bottomLeading
)

ConsoleDock.start(configuration: configuration)
```

Shape:

```swift
public enum ConsoleDock.FloatingButtonPosition: Equatable {
    case topLeading
    case topTrailing
    case bottomLeading
    case bottomTrailing
}
```

`ConsoleDock.Configuration` gains:

```swift
public var floatingButtonPosition: ConsoleDock.FloatingButtonPosition
```

Default remains `.bottomTrailing`, matching the current behavior.

Objective-C/Core:

```objc
typedef NS_ENUM(NSInteger, CDKFloatingButtonPosition) {
    CDKFloatingButtonPositionTopLeading = 0,
    CDKFloatingButtonPositionTopTrailing = 1,
    CDKFloatingButtonPositionBottomLeading = 2,
    CDKFloatingButtonPositionBottomTrailing = 3,
};

@property (nonatomic, assign) CDKFloatingButtonPosition floatingButtonPosition;
```

The property belongs on `CDKConfiguration` so Objective-C apps can configure the bundled UIKit trigger through the existing start path.

### Floating Button Runtime Control

Swift:

```swift
ConsoleDock.hideFloatingButton()
ConsoleDock.showFloatingButton()
```

Objective-C/UIKit facade:

```objc
[CDKConsoleDockUIKit hideFloatingButton];
[CDKConsoleDockUIKit showFloatingButton];
```

Behavior:

- These APIs only affect the bundled floating trigger.
- They do not start or stop ConsoleDock.
- They are no-ops when ConsoleDock is not running.
- `ConsoleDock.showConsole()` and `[CDKConsoleDockUIKit showConsole]` can present the console without making the floating trigger visible when the app started with `showsFloatingButton = false`.
- If the trigger is shown after being hidden, it appears at the configured starting position unless the current runtime already has a dragged button instance.

## UIKit UI Changes

### Floating Trigger

- Use the configured start position when the overlay window is installed.
- Continue supporting drag within the overlay bounds.
- Keep the button visual style and accessibility identifier unchanged.
- Do not persist dragged position across launches or restarts in this release.

### Logs Jump Menu

Add a compact `Jump` control in the Logs view with an action sheet:

- `Latest Visible Log`
- `First Visible Error`

Behavior:

- `Latest Visible Log` scrolls to the last currently visible entry and is disabled when no entry is visible.
- `First Visible Error` scrolls to the first currently visible `.error` or `.fault` entry and is disabled when the current visible set has no error/fault entry.
- The menu respects current search, source, and level filters.
- It does not change filters or pause/resume state.

### Actions Search

Add a search controller for the Actions page.

Behavior:

- Search matches action `id`, `title`, `group`, and `detail`, case-insensitively.
- Grouping is rebuilt from the filtered set.
- Empty state distinguishes between no registered actions and no actions matching the current search.
- Disabled and destructive metadata keeps the existing display behavior.
- Search is local-only UI filtering. It does not execute actions and does not persist query state.

## Tests And Validation

Unit tests:

- Swift configuration defaults include `.bottomTrailing`.
- Swift configuration maps `floatingButtonPosition` into `CDKConfiguration`.
- `CDKConfiguration.defaultConfiguration` exposes `CDKFloatingButtonPositionBottomTrailing`.
- Objective-C/UIKit facade runtime trigger methods are safe no-ops before startup.
- Actions search filtering matches id, title, group, and detail through a small testable helper.

UI and smoke tests:

- Swift sample can open the Logs jump menu and sees both jump actions.
- Jump latest and first error/fault work without crashing and keep the visible log list available.
- Swift and Objective-C samples can search Actions for a registered action and still execute it.
- Existing log detail, share, marker, pause/resume, clear, and action confirmation smoke checks continue passing.

Release validation:

- Existing `scripts/validate-release.sh` remains the release gate.
- Swift and Objective-C API surface validators include the new public APIs.
- UI accessibility identifier validator includes new Jump and Actions search identifiers.
- Versioned docs and roadmap move from `v0.5.0` to `v0.6.0` only after implementation and validation.

## Acceptance Criteria

- The project still starts with the same default visible floating button at bottom trailing.
- Apps can choose a different initial floating button corner.
- Apps can hide/show the floating trigger without stopping logging.
- Manual console presentation does not force the floating trigger to appear when it was disabled by configuration.
- Testers can jump to the latest visible log and first visible error/fault from the Logs view.
- Testers can search Debug Actions by id, title, group, or detail.
- Swift and Objective-C samples demonstrate the new capabilities.
- Local release validation and tag release validation pass before any public release.
