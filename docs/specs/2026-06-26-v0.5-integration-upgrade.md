# ConsoleDock v0.5 Integration Upgrade

Status: accepted design specification for the v0.5.0 implementation phase.

## Summary

ConsoleDock v0.5.0 should make real existing iOS projects easier to adopt without changing the product into a platform, network inspector, crash reporter, or distribution project.

The release builds on v0.4.0. ConsoleDock already has stdout/stderr capture, explicit native logging, Debug Actions, log detail, local issue reports, Swift and Objective-C samples, and release validation. The remaining adoption gap is that older apps often have a central logger, macro, or third-party logging destination and need a clear, low-risk way to add ConsoleDock as one more local debug sink.

v0.5.0 focuses on three connected improvements:

1. Lightweight log forwarders for Swift and Objective-C migration points.
2. Small issue-report/export polish that stays local and user initiated.
3. Bounded Debug Actions refinements for disabled and destructive actions.

## Goals

- Reduce old-project adoption friction by giving apps a reusable forwarding object instead of forcing every integration guide to hand-write `ConsoleDock.log(level:message:)` calls.
- Keep the existing logger as the source of truth; ConsoleDock should become an additional local destination.
- Improve Objective-C migration clarity for macro/function/logger-class integration.
- Make issue reports easier to copy or generate from app-owned debug surfaces without adding default persistence or upload.
- Add a small Debug Actions metadata upgrade that helps test panels represent disabled and destructive actions accurately.
- Keep SPM-only source distribution and the local-only debug/test boundary.

## Non-Goals

- CocoaPods, podspec publication, or CocoaPods trunk work.
- XCFramework or binary artifact generation.
- Network request inspection.
- Crash reporting or crash capture.
- Remote command delivery, remote issue creation, or upload.
- Default disk persistence.
- SwiftUI rewrite.
- Automatic route discovery.
- Automatic, complete Swift `Logger`, `os_log`, or Apple unified logging capture.
- A packaged adapter suite for CocoaLumberjack, XCGLogger, SwiftyBeaver, or other third-party frameworks before real demand selects one.

## Public API

### Swift Log Forwarder

Add a Swift app-facing forwarding helper in the `ConsoleDock` product:

```swift
let forwarder = ConsoleDock.LogForwarder(category: "AppLog", minimumLevel: .info)

AppLog.addSink { level, message in
    forwarder.log(level: level, message: message)
}
```

Shape:

```swift
public struct ConsoleDock.LogForwarder {
    public let category: String?
    public let minimumLevel: ConsoleDock.LogLevel

    public init(category: String? = nil, minimumLevel: ConsoleDock.LogLevel = .debug)

    public func log(level: ConsoleDock.LogLevel, message: String)
    public func debug(_ message: String)
    public func info(_ message: String)
    public func warning(_ message: String)
    public func error(_ message: String)
    public func fault(_ message: String)
}
```

Behavior:

- `minimumLevel` drops lower-priority messages before they enter ConsoleDock.
- `category` is optional and becomes a safe prefix in the stored message, for example `[AppLog] message`.
- The forwarder calls existing ConsoleDock native logging, so redaction, truncation, notifications, detail, copy, and reports behave exactly like other native entries.
- The forwarder does not start ConsoleDock and does not persist messages.
- Logging while ConsoleDock is not running follows existing native logging behavior.

### Objective-C Log Forwarder

Add an Objective-C-compatible helper in `ConsoleDockCore`:

```objc
CDKLogForwarder *forwarder = [[CDKLogForwarder alloc] initWithCategory:@"AppLog"
                                                          minimumLevel:CDKLogLevelInfo];
[forwarder logWithLevel:CDKLogLevelInfo message:message];
```

Shape:

```objc
@interface CDKLogForwarder : NSObject

@property (nonatomic, copy, readonly, nullable) NSString *category;
@property (nonatomic, assign, readonly) CDKLogLevel minimumLevel;

- (instancetype)initWithCategory:(nullable NSString *)category
                    minimumLevel:(CDKLogLevel)minimumLevel NS_DESIGNATED_INITIALIZER;
- (instancetype)init;

- (void)logWithLevel:(CDKLogLevel)level message:(NSString *)message;
- (void)debug:(NSString *)message;
- (void)info:(NSString *)message;
- (void)warning:(NSString *)message;
- (void)error:(NSString *)message;
- (void)fault:(NSString *)message;

@end
```

Behavior matches the Swift forwarder. It is intentionally a small helper around `CDKConsoleDock logWithLevel:message:` rather than a framework-specific adapter.

### Issue Report Text Access

Expose issue-report generation for app-owned debug surfaces:

```swift
let report = ConsoleDock.issueReportText()
```

Objective-C/UIKit facade:

```objc
NSString *report = [CDKConsoleDockUIKit issueReportText];
```

Behavior:

- Returns the same local plain-text report currently used by the bundled UIKit `Share Issue Report` action.
- Includes session metadata, diagnostics, marker index, and all currently retained redacted logs.
- Does not write files, upload, or create issues.
- Allows custom debug panels to copy or attach the report through app-owned user actions.

### Debug Action Metadata

Add bounded metadata to Debug Actions:

```swift
ConsoleDock.registerAction(
    id: "reset.checkout",
    title: "Reset Checkout",
    group: "State",
    detail: "Clears local checkout state.",
    requiresConfirmation: true,
    isEnabled: checkoutDebugState.canReset,
    style: .destructive
) {
    CheckoutDebugState.reset()
}
```

Swift shape:

```swift
public enum ConsoleDock.DebugActionStyle {
    case normal
    case destructive
}
```

Extend the existing registration method with defaulted `isEnabled` and `style` parameters. Existing Swift call sites remain source-compatible.

Objective-C/UIKit shape:

- Keep the existing `CDKConsoleDockUIKit registerActionWithIdentifier:title:group:detail:requiresConfirmation:handler:` method.
- Add a longer method with `isEnabled` and `style`.
- Add an Objective-C-compatible style enum in the UIKit facade.

Behavior:

- Disabled actions remain visible in the Actions list, use disabled row styling, cannot be selected, and log a skipped message if performed programmatically.
- `style = destructive` affects UI wording/styling and confirmation button style only. It does not imply confirmation by itself.
- Existing `requiresConfirmation` behavior remains unchanged.
- Handlers still run on the main thread and outside the registry lock.

## UIKit UI Changes

### Logs Share Menu

The bundled share action sheet should add:

- `Copy Issue Report`
- existing `Share Visible Logs`
- existing `Share All Logs`
- existing `Share Issue Report`

`Copy Issue Report` copies the same report text to the pasteboard. It does not write a file.

### Actions List

Rows should reflect:

- disabled state;
- destructive style;
- confirmation requirement.

Disabled rows should be visually de-emphasized and should not execute on tap. Destructive rows should show a clear detail line such as `Destructive` and use a destructive confirmation button only when confirmation is required.

Document the macro/function migration path with the forwarder:

```objc
static CDKLogForwarder *AppLogForwarder(void) {
    static CDKLogForwarder *forwarder;
    static dispatch_once_t onceToken;
    dispatch_once(&onceToken, ^{
        forwarder = [[CDKLogForwarder alloc] initWithCategory:@"AppLog" minimumLevel:CDKLogLevelDebug];
    });
    return forwarder;
}
```

## Tests And Validation

Unit tests:

- Swift forwarder prefixes category and forwards at the mapped level.
- Swift forwarder drops entries below `minimumLevel`.
- Objective-C `CDKLogForwarder` forwards and filters levels.
- Forwarders use the existing redaction and truncation path.
- `ConsoleDock.issueReportText()` matches the issue report formatter.
- `CDKConsoleDockUIKit issueReportText` returns report text.
- Debug Action registration stores `isEnabled` and `style`.
- Duplicate Debug Action id replacement updates `isEnabled` and `style` without duplicating rows.
- Disabled Debug Actions do not run handlers when performed programmatically.
- Disabled Debug Actions do not hold registry locks while skipped.
- Destructive style does not force confirmation.

UI / smoke tests:

- Swift sample `App logger sink` still generates expected ConsoleDock entries.
- Objective-C sample `App logger sink` still generates expected ConsoleDock entries.
- Share menu exposes `Copy Issue Report` and `Share Issue Report`.
- Actions list exposes disabled/destructive metadata enough for UI automation to assert the row exists without executing disabled actions.

Release validation:

- Existing `scripts/validate-release.sh` continues to pass.
- Swift API surface validator includes the new Swift forwarder, issue report text, and Debug Action style metadata.
- Objective-C API surface validator includes `CDKLogForwarder`, issue report text, and the extended Debug Action registration method.
- Sample validation and UI accessibility identifier validation include the new copy/report/action identifiers.

## Completion Criteria

v0.5.0 is complete when:

- Swift and Objective-C forwarders exist, are tested, and are used by the sample app logger sink paths.
- Issue report text can be copied from the bundled UI and generated through public Swift and Objective-C/UIKit APIs.
- Debug Actions support disabled and destructive metadata without breaking existing registration calls.
- Docs, samples, validators, and roadmap describe v0.5.0 accurately.
- Full local release validation passes.
- `main`, tag `v0.5.0`, GitHub Release, Release Validation workflow, and post-release verifier all pass before marking the goal complete.
