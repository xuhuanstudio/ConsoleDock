# ConsoleDock v0.10 Session Timeline

`v0.10.0` makes the current local debug session easier to understand at a glance. ConsoleDock already stores markers, Debug Action execution history, retained error/fault logs, App Context, and issue reports. The next useful step is to expose those existing signals as a bundled UIKit Timeline view instead of forcing testers to reconstruct the sequence from the full Logs list or exported report text.

## Goals

- Add a local-only Timeline view to the bundled UIKit panel.
- Aggregate manual markers, local Debug Action executions, and retained error/fault log entries into one timestamp-ordered sequence.
- Reuse the same timeline builder for the UIKit Timeline view and issue-report reproduction timeline text.
- Let testers open useful detail from timeline rows without introducing a new public query language or persistence layer.
- Keep the release focused on current-session triage, not long-term history or remote automation.

## Non-Goals

- Default disk persistence of logs, timeline events, action history, parameters, context, or reports.
- Remote upload, remote commands, issue creation, or automation-platform behavior.
- Network inspection, crash reporting, automatic route discovery, or debugger features.
- CocoaPods, XCFramework, binary artifacts, or new distribution channels.
- A public Timeline API in `v0.10.0`; the first release keeps the model internal until real integration demand appears.
- Timeline filtering, regex, date ranges, saved views, or editable timeline annotations.

## Timeline Events

The internal timeline model has three event kinds:

- `marker`: a native info log entry whose message starts with `[marker]`.
- `action`: one `ConsoleDock.DebugActionExecution` from the current process session.
- `log`: a retained log entry whose level is `error` or `fault`.

Events are sorted by timestamp ascending. When timestamps match, builder insertion order is stable so repeated report generation and UI reloads do not reorder equal-time events.

Each event exposes internal metadata for UI and text formatting:

- stable internal id;
- timestamp;
- kind label;
- title;
- optional subtitle;
- optional detail text;
- optional related `ConsoleDock.LogEntry`;
- optional related `ConsoleDock.DebugActionExecution`;
- severity color hint for log/action outcome styling.

`warning` entries are intentionally excluded from the first Timeline view because warnings can be noisy in real apps. They remain visible in Logs and issue-report full logs.

## UIKit Timeline View

The bundled panel adds a Timeline mode alongside Logs, Actions, and Context. The exact segmented control title should be `Timeline`.

The Timeline view:

- displays an empty state: `No timeline events yet.`;
- shows marker, action, and error/fault rows in timestamp order;
- uses compact rows with timestamp, kind/outcome, title, and short detail;
- refreshes when ConsoleDock entries change or Debug Actions change;
- provides a Refresh navigation action for manual reloads;
- opens log detail for marker/error/fault rows by pushing the existing `ConsoleDockLogDetailViewController`;
- opens a lightweight action detail screen for action rows, with copy support for the formatted action event;
- remains UIKit-only and private to the `ConsoleDock` target.

The view does not pause live updates separately from Logs. It always renders a fresh current-session snapshot because the timeline is for high-level triage, not detailed log reading.

## Issue Report Reuse

`ConsoleDockIssueReportFormatter` should stop owning separate private timeline assembly logic. It should call the shared internal timeline builder and format those events into the existing `Reproduction Timeline:` section.

The report's public text shape should stay compatible:

- marker lines remain labeled `[marker]`;
- action lines remain labeled `[action]` with outcome;
- error/fault log lines remain labeled `[log]`;
- empty timelines still render `  (no timeline events)`.

Small wording improvements are acceptable, but the report must continue to include session metadata, diagnostics, App Context, markers, and full logs.

## Tests

Unit tests should cover:

- timeline builder returns no events for empty input;
- marker, action, and error/fault log events are included;
- info/debug/warning logs are excluded unless they are markers;
- events sort by timestamp with stable equal-time ordering;
- issue report reproduction timeline uses the shared builder output;
- action event formatting includes outcome, action id/title, group, parameter summary, and message when present.

UI smoke coverage should verify in both Swift and Objective-C sample apps:

- the panel exposes a Timeline mode;
- the Timeline table appears;
- marker, action, and error/fault entries appear after the existing smoke flow creates them;
- tapping a marker or error/fault timeline row opens log detail;
- tapping an action timeline row opens action detail with copy controls.

Validators should add stable accessibility identifiers for the Timeline table, empty state, refresh button, action detail text, and action detail copy button.

## Documentation

Documentation updates should be minimal and versioned:

- README and Chinese README mention `v0.10.0` as the current release and describe Session Timeline in one short section.
- DocC adds a short Session Timeline page or updates Test Session Reports if that is cleaner.
- Roadmap records `v0.10 - Session Timeline`.
- Changelog records `v0.10.0` under a dated heading only when the release is ready.
- Sample walkthrough includes Timeline smoke verification.

Docs must keep the existing boundaries explicit: Timeline is current-session, local-only, user-visible debugging assistance. It is not persistent history, telemetry, a remote test runner, or a replacement for Xcode Console.
