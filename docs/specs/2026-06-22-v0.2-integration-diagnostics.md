# ConsoleDock v0.2 Integration Diagnostics

Date: 2026-06-22

Status: accepted design for the next implementation phase.

## Objective

ConsoleDock v0.2 should make real app integration easier to verify. A developer or tester should be able to answer, from code or the bundled UIKit console, whether ConsoleDock is running, which capture paths are configured, how many entries are stored, what limits are active, and whether the current build is gated by Release safety.

This phase does not change ConsoleDock's product boundary. ConsoleDock remains an in-app debug console for iOS testing, with local in-memory storage, redaction before storage, Release disabled by default, stdout/stderr capture as the only baseline zero-intrusion path, and explicit APIs or adapters required for reliable Swift `Logger`, `os_log`, or Apple unified logging visibility.

## Current Context

The `v0.1.0` codebase already includes:

- `ConsoleDockCore` Objective-C/C-compatible lifecycle, storage, redaction, line framing, and stdout/stderr file descriptor capture.
- Swift facade APIs over the core.
- UIKit floating button and console panel with search, source and level filters, pause/resume live follow, selected-entry copy, clear, and share/export for the visible in-memory snapshot.
- Swift and Objective-C sample apps.
- release validation covering SwiftPM, tests, Release safety, DocC, sample builds, source archives, and governance metadata.

The next gap is operational clarity. If a host app integration does not show logs, users currently need to infer state from side effects. A small diagnostic surface gives them a supported way to inspect the active configuration and store state without reading internals.

## User-Facing Requirements

### Core Diagnostics

`ConsoleDockCore` should expose an Objective-C-compatible immutable diagnostics object named `CDKDiagnostics`.

It should include:

- `running`: whether `CDKConsoleDock` is currently running.
- `captureStandardOutput`: whether stdout capture is enabled in the effective active configuration.
- `captureStandardError`: whether stderr capture is enabled in the effective active configuration.
- `showsFloatingButton`: whether the active configuration requested the bundled UIKit floating button.
- `allowsReleaseBuilds`: whether the active runtime configuration allows Release startup when compiled with `CONSOLEDOCK_ENABLE_RELEASE`.
- `maximumEntries`: the configured in-memory entry cap.
- `maximumMessageLength`: the configured stored-message cap.
- `entryCount`: the current number of stored entries.
- `redactedEntryCount`: the number of stored entries marked redacted.
- `truncatedEntryCount`: the number of stored entries marked truncated.
- `partialEntryCount`: the number of stored entries marked partial.

When ConsoleDock is not running, diagnostics should still be available. If a previous session's in-memory store is still retained after `stop`, diagnostics should report the current store counts and `running = false`. If ConsoleDock has never started or no store exists, diagnostics should report zero counts and default configuration values. This matches the current lifecycle, where `stop` restores capture state without clearing readable entries.

### Swift Facade

The Swift target should expose a small value type, `ConsoleDock.Diagnostics`, with the same fields using Swift naming and `Int` counts.

Public API shape:

```swift
public struct Diagnostics: Equatable {
    public let isRunning: Bool
    public let capturesStandardOutput: Bool
    public let capturesStandardError: Bool
    public let showsFloatingButton: Bool
    public let allowsReleaseBuilds: Bool
    public let maximumEntries: Int
    public let maximumMessageLength: Int
    public let entryCount: Int
    public let redactedEntryCount: Int
    public let truncatedEntryCount: Int
    public let partialEntryCount: Int
}

public static var diagnostics: Diagnostics { get }
```

Swift should remain a thin wrapper over core diagnostics. It should not recompute state from a separate implementation except for type conversion.

### UIKit Console

The bundled UIKit console should show compact diagnostic state without turning into a settings screen.

Recommended UI:

- Add a small status header above the table, below the level segmented control.
- Show: `Running`, `Entries`, `stdout`, `stderr`, and `Limits`.
- Keep text concise and monospaced enough to match the existing console style.
- Update the header when entries refresh, filters change, live updates pause/resume, clear is tapped, start/stop changes are observed, or the panel appears.
- The share/export snapshot should include a diagnostics block before entries, so bug reports preserve capture state.

The status header should not include secrets, raw environment data, bundle identifiers, device identifiers, or filesystem paths.

### Snapshot Export

`ConsoleDockSnapshotFormatter` should accept optional diagnostics and visible-entry metadata.

Recommended output shape:

```text
ConsoleDock Log Snapshot
Generated: 2026-06-22T00:00:00.000Z
Entries: 12
Visible Entries: 5
Diagnostics:
  Running: true
  stdout: enabled
  stderr: enabled
  Limits: entries=2000 messageLength=8192
  Redacted: 2
  Truncated: 1
  Partial: 0

[timestamp] [source] [LEVEL] message
```

Existing callers that do not pass diagnostics should keep the current output format unless a deliberate test update documents the new default. This keeps the formatter easy to adopt outside the bundled UI.

## Internal Design

### Core State

`CDKConsoleDock` already owns the authoritative process-global state:

- running/stopping flags;
- effective copied configuration;
- in-memory entries;
- stdout/stderr capture object.

Diagnostics should be built under the same synchronization boundary used by `entries`, `start`, `stop`, and `appendEntryWithLevel`. The snapshot should copy only scalar state and counts while the lock is held, then return an immutable object.

The implementation should avoid exposing `CDKStandardOutputCapture` internals. Capture enabled/disabled should come from the effective configuration, not from file descriptor implementation details.

### Entry Counts

Counts should be derived from the stored entries:

- `entryCount`: total stored entries.
- `redactedEntryCount`: entries where `redacted == YES`.
- `truncatedEntryCount`: entries where `truncated == YES`.
- `partialEntryCount`: entries where `isPartial == YES`.

These counts reflect the current bounded store, not all historical logs emitted during the app session. Documentation should call this out.

### Notification Behavior

No new notification name is required for v0.2. Existing `entriesDidChange` notifications are sufficient for UI updates driven by entry changes.

The UIKit panel should also refresh diagnostics on:

- `viewWillAppear`;
- pause/resume toggle;
- clear action;
- explicit show action.

If a future version adds configuration changes without entry changes, a separate diagnostics notification can be considered then.

## Error Handling

Diagnostics should be best-effort and non-throwing:

- Core and Swift diagnostics APIs should always return a value.
- If no active configuration exists, use `CDKConfiguration.defaultConfiguration` as the configuration baseline.
- If counts are unavailable because the store is nil, return zero counts.
- Release safety state should be represented by the runtime `allowsReleaseBuilds` flag. Compile-time availability remains documented in Release safety docs and tests.

## Testing

Add focused tests rather than broad UI automation:

- core diagnostics before start;
- core diagnostics after start with custom configuration;
- counts for redacted, truncated, and partial entries;
- diagnostics after clear;
- diagnostics after stop preserves store counts while marking not running;
- Swift diagnostics maps every core field correctly;
- snapshot formatter includes diagnostics when provided and preserves old output when omitted;
- UIKit-facing header formatting can be covered through a small pure helper if the UI text is separated from `UIViewController`.

Run the full release validation script after implementation:

```sh
scripts/validate-release.sh
```

## Documentation

Update:

- `README.md` and `README.zh-CN.md` with a short diagnostics example.
- DocC Getting Started or a dedicated diagnostics page.
- `docs/sample-app-walkthrough.md` to include checking the status header and snapshot diagnostics.
- `CHANGELOG.md` under `Unreleased`.

Documentation must keep the logging boundary honest. Diagnostics should not be framed as proof that Swift `Logger`, `os_log`, Apple unified logging, other-process logs, sanitizer diagnostics, LLDB output, or Xcode-only output are captured.

## Non-Goals

This phase does not add:

- CocoaPods podspec.
- XCFramework automation.
- third-party logger adapters.
- persistent export files.
- disk-backed logging.
- remote upload.
- network inspection.
- crash reporting.
- system log reading.
- automatic Swift `Logger` / `os_log` ingestion.

## Acceptance Criteria

The phase is complete when:

- Objective-C callers can read `CDKConsoleDock.diagnostics`.
- Swift callers can read `ConsoleDock.diagnostics`.
- The UIKit console shows compact diagnostic state.
- Share/export can include diagnostics for the visible snapshot.
- Tests cover core, Swift mapping, and formatter behavior.
- README, DocC or docs, and changelog describe the capability and boundary.
- `scripts/validate-release.sh` passes locally and in GitHub Actions after the implementation commit is pushed.
