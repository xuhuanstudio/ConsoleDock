# ConsoleDock v0.13.0 Integration Doctor

## Summary

`v0.13.0` adds a local Integration Doctor so a developer or tester can understand the current ConsoleDock integration state without connecting Xcode or reading source code. The feature should answer practical adoption questions:

- Is ConsoleDock running?
- Which capture paths are enabled?
- Which sources have produced retained entries?
- Are entries being redacted, truncated, or flushed as partial lines?
- Are Debug Actions, App Context, and Local Session Archives present?
- What should the app check when expected logs do not appear?

This is a functional debugging aid, not a new distribution channel or a remote diagnostics system.

## Scope

Deliver:

- A generated Integration Diagnosis text report.
- A `ConsoleDock.integrationDiagnosisText()` Swift API.
- A `CDKConsoleDockUIKit.integrationDiagnosisText` Objective-C/UIKit facade API.
- A ConsoleDock-owned `ConsoleDock Health` section in the existing `Context` tab.
- A `Copy Integration Diagnosis` button in the `Context` tab.
- Issue reports that include the same health section near diagnostics.
- Unit tests for source counts, recommendation text, formatter stability, and public facades.
- Swift sample UI smoke coverage that opens `Context` and verifies the health section.
- README, DocC, roadmap, changelog, sample walkthrough, and API validator updates.

Do not deliver:

- A new tab or large UIKit redesign.
- Automatic repair or configuration mutation.
- Reading Swift `Logger`, `os_log`, or Apple unified logging entries.
- Remote upload, remote issue creation, network inspection, crash reporting, CocoaPods, or XCFramework distribution.
- A stable public health model with many typed fields.

## Data Model

Keep the public API small. The stable public output for this release is text:

```swift
let text = ConsoleDock.integrationDiagnosisText()
```

Objective-C/UIKit integrations use:

```objc
NSString *text = [CDKConsoleDockUIKit integrationDiagnosisText];
```

Internally, build a small snapshot from existing sources:

- `ConsoleDock.diagnostics`
- `ConsoleDock.entries`
- registered Debug Actions
- action execution history
- App Context provider presence and normalized snapshot size
- Local Session Archive count when available
- `ConsoleDock.sessionMetadata`

The internal snapshot may change during `v0.x`; it should not be documented as a stable structured API.

## Diagnosis Content

The text report should contain:

- title and generated timestamp;
- session identifier and start state;
- running state and capture configuration;
- retained entry totals by source and level;
- redacted, truncated, and partial counts;
- Debug Action registration and execution counts;
- App Context provider and item counts;
- Local Session Archive count or archive error;
- concise local recommendations.

Recommendations should be deterministic and practical. Examples:

- if not running, say to call `ConsoleDock.start()` in a debug/test build and check Release safety;
- if no native entries, suggest explicit `ConsoleDock.info/error` calls or logger forwarders;
- if stdout/stderr capture is disabled, say those descriptor paths are disabled by configuration;
- if stdout/stderr capture is enabled but no descriptor entries exist, mention flushing and direct descriptor writes;
- if the user expects Swift `Logger`, `os_log`, or unified logging, state that ConsoleDock does not promise full capture and recommend explicit forwarding;
- if no Debug Actions are registered, state that actions are app-registered local shortcuts, not auto-discovered routes;
- if no App Context provider is registered, suggest registering only safe local diagnostics.

## UIKit Integration

Reuse the existing `Context` tab.

- Prepend a ConsoleDock-owned `ConsoleDock Health` section before app-provided context sections.
- Keep the app-provided sections unchanged.
- Add a copy button to the Context navigation items.
- The copy button writes `ConsoleDock.integrationDiagnosisText()` to the pasteboard.
- Empty state should only show when there is neither health nor app context. In practice, the health section means the Context tab is usually non-empty.

The UI should stay compact and readable with the current dark UIKit styling.

## Issue Reports

Issue reports already include diagnostics and app context. Add a `ConsoleDock Health` block after diagnostics and before app context so copied/saved/shared reports include the same integration cues.

Keep this text local-only. It must not upload, persist raw logs, or claim complete system logging.

## Testing

Unit tests:

- formatter reports source and level counts;
- formatter emits not-running guidance;
- formatter emits native/log-forwarder guidance when no native entries exist;
- formatter emits descriptor capture guidance for stdout/stderr states;
- formatter includes Debug Actions, App Context, and archive state;
- `ConsoleDock.integrationDiagnosisText()` includes current entries;
- Objective-C/UIKit facade returns the same diagnosis text shape;
- issue report includes `ConsoleDock Health`.

UI smoke:

- Swift sample opens `Context`;
- verifies `ConsoleDock Health`;
- verifies at least one health item such as `Running` or `Entry Sources`;
- verifies `Copy Integration Diagnosis` is present.

Validators:

- Swift API surface includes `ConsoleDock.integrationDiagnosisText()`;
- Objective-C/UIKit API surface includes `CDKConsoleDockUIKit.integrationDiagnosisText`;
- accessibility identifier validation includes the copy button.

## Release Criteria

- `swift test` passes.
- Swift and Objective-C sample UI smoke scripts pass.
- `scripts/validate-release.sh` passes with `CONSOLEDOCK_RUN_UI_SMOKE=1`.
- Public release notes repeat the boundary that ConsoleDock is not a full Xcode Console or Apple unified logging replacement.
