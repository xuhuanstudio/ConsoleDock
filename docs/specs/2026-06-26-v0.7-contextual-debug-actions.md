# ConsoleDock v0.7.0 Contextual Debug Actions

## Summary

ConsoleDock v0.7.0 should make Debug Actions useful for real local test flows that need small pieces of input and should make issue reports carry app-owned context. The release remains local-only, debug/test-first, UIKit-based, and SPM-only.

The release adds two bounded capabilities:

1. Parameterized Debug Actions.
2. App Context snapshots for panel display and issue reports.

This is not a move toward a remote command system, route automation, a persistence layer, a crash reporter, or a network inspector.

## Goals

- Let an app register local Debug Actions that ask the tester for a small set of typed parameters before running.
- Keep parameter collection simple enough for a UIKit alert/form flow.
- Execute action handlers on the main thread with a normalized parameter payload.
- Log action started, completed, skipped, and failed entries with enough context to diagnose local testing.
- Let the app register local key-value context that appears in the bundled panel and issue reports.
- Keep all context local, app-provided, bounded, and generated only when requested.
- Maintain Swift and Objective-C integration surfaces.

## Non-Goals

- Remote commands or remotely delivered action definitions.
- Automatic route discovery, reflection, page scanning, or AppRouter ownership.
- Parameter persistence, recent value storage, favorites, or saved presets.
- Complex async action state management.
- Arbitrary custom UIKit controls supplied by host apps.
- Nested objects, file pickers, image pickers, date pickers, secret fields, or multi-step forms.
- Network inspection, crash reporting, CocoaPods, XCFramework, SwiftUI rewrite, or default disk persistence.

## Parameterized Debug Actions

### Parameter Model

Parameter definitions are in-memory metadata stored with each registered Debug Action. The first version supports:

- string;
- number;
- bool;
- choice.

Each parameter has:

- `id`: required, single-line, unique within the action.
- `title`: required, single-line display label.
- `detail`: optional explanatory text.
- `isRequired`: whether empty input prevents execution.
- `defaultValue`: optional type-compatible value.

Choice parameters additionally have a non-empty list of choices. Each choice has:

- `id`: required single-line value passed to handlers.
- `title`: required single-line label.

Duplicate parameter ids inside an action are ignored after normalization; the first valid parameter wins. Invalid definitions are dropped rather than causing registration failure. If all parameters are invalid, the action behaves like a normal no-parameter action.

### Swift API

Existing registration APIs remain source-compatible.

New Swift-facing concepts:

```swift
ConsoleDock.DebugActionParameter
ConsoleDock.DebugActionParameterValue
ConsoleDock.DebugActionChoice
ConsoleDock.DebugActionParameters
```

Registration shape:

```swift
ConsoleDock.registerAction(
    id: "open.order",
    title: "Open Order",
    group: "Navigation",
    detail: "Jump to an order by id",
    parameters: [
        .string(id: "orderId", title: "Order ID", isRequired: true)
    ]
) { parameters in
    let orderID = parameters.string("orderId") ?? ""
    AppRouter.shared.openOrder(id: orderID)
}
```

The existing no-parameter closure overload remains:

```swift
ConsoleDock.registerAction(id: "smoke", title: "Generate Smoke Logs") {
    ConsoleDock.info("smoke")
}
```

### Objective-C API

Objective-C uses explicit factory methods and a dictionary payload:

```objc
CDKDebugActionParameter *parameter =
    [CDKDebugActionParameter stringParameterWithIdentifier:@"orderId"
                                                     title:@"Order ID"
                                                    detail:nil
                                                isRequired:YES
                                              defaultValue:nil];

[CDKConsoleDockUIKit registerActionWithIdentifier:@"open.order"
                                            title:@"Open Order"
                                            group:@"Navigation"
                                           detail:@"Jump to an order by id"
                             requiresConfirmation:NO
                                       parameters:@[ parameter ]
                                          handler:^(NSDictionary<NSString *, id> *parameters) {
    NSString *orderID = parameters[@"orderId"];
    [AppRouter openOrderWithIdentifier:orderID];
}];
```

Objective-C values are:

- `NSString` for string and choice;
- `NSNumber` for number and bool.

Existing Objective-C no-parameter APIs remain source-compatible.

### Execution Rules

- Actions and parameter metadata stay in memory only.
- Actions can still be registered before or after `ConsoleDock.start()`.
- Duplicate action ids replace the previous action without changing registration order.
- Parameterized handlers run on the main thread.
- Handlers are never executed while registry locks are held.
- Disabled actions do not show the parameter form and do not run.
- Confirmation runs before parameter input when both are present.
- If required parameters are missing or invalid, the UI does not execute the action.
- Programmatic execution accepts an optional parameter payload; without a payload, defaults are used where available and missing required values produce a skipped log entry.
- Swift throwing handlers write a failed log entry.
- Objective-C block handlers do not throw; app code records its own failures.

### Logging

ConsoleDock writes:

- `Debug action started: <title> [<id>]`
- `Debug action completed: <title> [<id>]`
- `Debug action failed: <title> [<id>] error=<error>`
- `Debug action skipped: <title> [<id>] disabled`
- `Debug action skipped: <title> [<id>] missing required parameters: <ids>`

Parameter values are not automatically written to logs. Apps can log their own parameterized behavior through existing ConsoleDock logging APIs after applying any app-specific privacy rules.

## UIKit Actions UI

The Actions list continues to group and search actions by id, title, group, and detail.

Tapping an enabled action behaves as follows:

1. If `requiresConfirmation` is true, show the existing confirmation prompt.
2. If the action has valid parameters, show a local parameter form.
3. If the action has no valid parameters, run immediately.

Form controls:

- string: text field;
- number: decimal keyboard text field with simple numeric parsing;
- bool: switch;
- choice: compact table/action-sheet style choice selector.

The form has `Cancel` and `Run Action`. It does not persist entered values. It should fit in the current bundled UIKit panel style and avoid making `ConsoleDockActionsViewController` much larger by moving parameter form logic into a private UIKit view controller/helper.

## App Context

### Purpose

App Context is app-owned diagnostic context for local testing. Examples:

- current user id;
- account, tenant, or store id;
- environment;
- feature flags;
- active experiment;
- last selected order id;
- app-specific diagnostics summary.

ConsoleDock does not discover this context. The host app must provide it explicitly.

### Model

App Context is a snapshot of sections and key-value items:

- section title;
- item key;
- item value.

The first version treats all values as strings after normalization. It does not support nested objects or privacy annotations. Apps should provide already-redacted values for anything sensitive.

### Swift API

```swift
ConsoleDock.setAppContextProvider {
    [
        ConsoleDock.AppContextSection(
            title: "App",
            items: [
                .init(key: "Environment", value: "staging"),
                .init(key: "User ID", value: currentUserID)
            ]
        )
    ]
}

ConsoleDock.clearAppContextProvider()
```

### Objective-C API

```objc
[CDKConsoleDockUIKit setAppContextProvider:^NSArray<CDKAppContextSection *> *{
    return @[
        [CDKAppContextSection sectionWithTitle:@"App"
                                         items:@[
            [CDKAppContextItem itemWithKey:@"Environment" value:@"staging"],
            [CDKAppContextItem itemWithKey:@"User ID" value:userID]
        ]]
    ];
}];
```

### Snapshot Rules

- Provider is optional.
- Provider is called on the main thread.
- Provider output is normalized into single-line keys and bounded multi-line values.
- Provider is not called while registry locks are held.
- Provider failures are not possible in Objective-C; Swift providers should not throw in v0.7.
- Context is included in issue reports at generation time.
- Context is visible in the bundled panel through a simple local Context view or section.
- Context is not persisted, uploaded, or automatically refreshed in the background.

## Issue Report Format

Issue reports add an `App Context` section after `Diagnostics` and before `Markers`.

Example:

```text
App Context:
  App:
    Environment: staging
    User ID: 12345
  Feature Flags:
    New Checkout: enabled
```

If no context provider is registered or it returns no valid items:

```text
App Context:
  (no app context)
```

## Compatibility

- Existing no-parameter Debug Action registrations remain valid.
- Existing Swift and Objective-C public APIs remain available.
- `ConsoleDockCore` logging storage remains unchanged.
- New v0.7 APIs live in the Swift facade and Objective-C/UIKit facade.
- `v0.x` still allows API iteration, but these APIs should be small and stable enough to guide `1.0`.

## Tests

Unit tests:

- parameter metadata normalization and invalid parameter dropping.
- duplicate parameter ids keep first valid parameter.
- no-parameter APIs remain compatible.
- parameterized Swift handler receives normalized values.
- missing required parameters skips execution and logs a skipped entry.
- default parameter values are used for programmatic execution.
- choice parameters reject invalid values.
- disabled action does not request parameters or execute.
- registry remains thread-safe and handlers run outside locks.
- App Context provider stores, clears, and snapshots sections.
- issue report includes App Context and handles empty context.

UI smoke tests:

- Swift sample opens Actions, runs a string-parameter action, and sees expected logs.
- Swift sample runs a bool or choice parameter action.
- Objective-C sample registers and runs at least one parameterized action.
- Context is visible or issue-report copy/share path includes App Context.
- Existing Logs, Actions search, confirmation, detail, copy, clear, and share flows still pass.

Release validation:

- `scripts/validate-release.sh` still passes.
- Swift API surface validator includes v0.7 APIs.
- Objective-C API surface validator includes v0.7 APIs.
- UI accessibility identifier validator includes parameter form/context identifiers.
- No new distribution channel gates.

## Success Criteria

- A host app can register an action requiring an order id and execute it from the device without adding custom test UI.
- A host app can include current local app context in a copied/shared issue report.
- Existing v0.6 no-parameter workflows remain unchanged.
- Full local release validation passes before tagging.
- GitHub Release Validation passes before publishing `v0.7.0`.
