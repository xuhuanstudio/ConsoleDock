# ConsoleDock v0.16 API Readiness

`v0.16` is a focused pre-1.0 API readiness pass. It should remove misleading public API semantics and tighten existing local reproduction behavior without expanding ConsoleDock into analytics, telemetry, remote upload, crash reporting, network inspection, CocoaPods, or XCFramework distribution.

## Goals

- Make marker handling explicit in stored entries so Timeline and issue reports do not infer markers from a user-visible message prefix alone.
- Remove the misleading Swift `Configuration: Equatable` conformance because the configuration owns an optional redactor closure that cannot be compared lawfully.
- Keep Swift and Objective-C/UIKit facade cleanup APIs aligned for local Debug Action execution history.
- Update API validators, tests, and docs so the next 1.0 review has fewer unresolved source-stability risks.

## Non-Goals

- No new remote transport, analytics, telemetry, or upload behavior.
- No change to SPM-only distribution.
- No SwiftUI rewrite or visual redesign.
- No new public query language or automation platform.
- No broad source-breaking API rename outside the specific `Configuration` equality correction.

## Design

### Marker Storage

`CDKLogEntry` gains an `isMarker` flag. `CDKConsoleDock.mark(_:)` stores entries with that flag set, while normal native logs remain non-marker entries even when their message starts with `[marker]`.

Swift `ConsoleDock.LogEntry` exposes the same `isMarker` flag. Existing public initializers keep defaulting to non-marker entries so fixture and app-created values remain source-compatible unless they explicitly need marker semantics.

Timeline and issue-report marker detection move from `native && message.hasPrefix("[marker]")` to the stored marker flag.

### Configuration Equality

`ConsoleDock.Configuration` no longer conforms to `Equatable`. The previous implementation compared only whether both configurations had a redactor, which could report two different redactor closures as equal. That behavior is not suitable for a stable 1.0 API.

### Objective-C/UIKit Action Cleanup

`CDKConsoleDockUIKit` gains `clearActionExecutionHistory`, matching the Swift facade. It clears only the current local execution history and does not unregister actions or clear recent parameter values.

## Validation

- Add unit tests proving normal `[marker]` native logs are not Timeline/report markers.
- Add unit tests proving `mark(_:)` sets the marker flag through the core and Swift facades.
- Add Objective-C/UIKit facade test coverage for `clearActionExecutionHistory`.
- Update Swift and Objective-C API surface validators for the changed public surface.
- Run full release validation before tagging.
