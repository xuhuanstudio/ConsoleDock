# ConsoleDock v0.11 Local Session Archive

`v0.11.0` adds a small local archive for test-session evidence. ConsoleDock already builds redacted issue reports with session metadata, diagnostics, App Context, reproduction timeline, markers, and retained logs. The next useful product step is to let a tester explicitly save one of those reports and reopen it after an app restart.

This is not a persistent logging database. The archive stores bounded report snapshots only after an app or user explicitly saves them.

## Goals

- Add an opt-in, local-only archive for recent issue-report snapshots.
- Persist already-redacted and already-truncated report text, not raw pre-redaction log streams.
- Keep archives bounded by count and report length.
- Expose minimal Swift APIs for saving, listing, deleting, and clearing archives.
- Expose Objective-C/UIKit facade APIs through `CDKConsoleDockUIKit`.
- Add bundled UIKit flows to save the current session report, review saved archives, copy an archive, share an archive, delete one archive, and clear all archives.
- Keep the release focused on tester evidence recovery after app restart.

## Non-Goals

- Default background persistence of every log entry.
- Crash recovery or guaranteed capture of the final process moments before termination.
- Remote upload, remote commands, issue creation, sync, or cloud storage.
- Network inspection, crash reporting, automatic route discovery, or debugger replacement behavior.
- Querying archived logs as a database, archive search, archive diffing, or saved filters.
- CocoaPods, XCFramework, binary artifacts, or new distribution channels.

## Archive Model

An archive is one saved issue-report snapshot with:

- stable archive id;
- creation date;
- source session id;
- source session start date when available;
- saved report title;
- optional note;
- entry count at save time;
- report character count before archive-level truncation;
- `isReportTruncated` flag;
- persisted report text.

Archives are sorted newest first for public APIs and UI.

The saved text should come from the same issue-report pipeline as `ConsoleDock.issueReportText()`. That keeps all existing redaction, truncation, timeline, and context behavior in one place. If the archive length cap is reached, the saved report text should end with an explicit truncation notice so a reader can tell the report was shortened for local archive storage.

## Persistence

Persistence is explicit:

- `ConsoleDock.saveSessionArchive(...)` saves the current session report.
- The bundled UIKit share menu adds `Save Session Archive`.
- Sample apps can register a Debug Action that calls the same API.

ConsoleDock should not automatically persist logs or reports in the background for `v0.11.0`. A user-triggered save is the correct first version because it is easier to reason about privacy and disk usage.

The storage location should be app-local and private to the app sandbox. The implementation should prefer Application Support when available, create a `ConsoleDock/SessionArchives` directory, and write JSON atomically. Tests should be able to inject a temporary directory.

Defaults:

- maximum archives: `5`
- maximum stored report characters: `256_000`

Invalid or very small limits should be normalized internally so the archive can still save a coherent truncation notice and keep at least one archive.

## Swift API

Public Swift API should stay in `ConsoleDock.swift`:

```swift
let archive = try ConsoleDock.saveSessionArchive(note: "Checkout smoke test")
let archives = try ConsoleDock.sessionArchives()
try ConsoleDock.deleteSessionArchive(id: archive.id)
try ConsoleDock.clearSessionArchives()
```

The archive model should be a value type nested under `ConsoleDock`, for example `ConsoleDock.SessionArchive`. It should expose metadata and `reportText`.

`v0.x` allows API iteration, but this surface should stay small because archive storage is likely to remain part of the product.

## Objective-C/UIKit API

Objective-C access should stay in `ConsoleDockUIKit.swift`:

- `CDKSessionArchive`
- `[CDKConsoleDockUIKit saveSessionArchiveWithNote:error:]`
- `[CDKConsoleDockUIKit sessionArchivesWithError:]`
- `[CDKConsoleDockUIKit deleteSessionArchiveWithIdentifier:error:]`
- `[CDKConsoleDockUIKit clearSessionArchivesWithError:]`

Objective-C callers do not need direct access to storage internals. They should receive immutable wrapper objects containing archive metadata and report text.

## UIKit Flow

The existing Logs share menu is the right entry point. It should add:

- `Save Session Archive`
- `Saved Session Archives`

The saved archives screen should:

- show `No saved session archives.` when empty;
- list archives newest first;
- show title, date, entry count, session id, optional note, and truncation state;
- open an archive detail screen on tap;
- offer `Clear All Archives` with confirmation.

The archive detail screen should:

- show archive metadata and full saved report text in a scrollable text view;
- provide `Copy Archive`;
- provide `Share Archive`;
- provide `Delete Archive` with confirmation.

This UI remains private UIKit. No SwiftUI rewrite is included.

## Tests

Unit tests should cover:

- saving an archive stores current issue-report text and metadata;
- archives are returned newest first;
- maximum archive count prunes oldest archives;
- maximum report length truncates text and records `isReportTruncated`;
- archive text uses redacted issue-report output rather than raw secret text;
- delete removes only the requested archive;
- clear removes all archives;
- storage survives a new store instance pointing at the same directory;
- malformed archive files do not crash listing;
- Objective-C/UIKit facade methods bridge archive save/list/delete/clear.

UI smoke coverage should verify:

- Swift sample app can open the share menu and see archive actions;
- saving the current session shows a saved archive row;
- tapping an archive opens detail with copy/share/delete controls;
- clearing archives confirms and returns to the empty state;
- Objective-C sample app exposes the archive menu actions at least once.

Validators should add stable accessibility identifiers for the archive menu actions, archives table, empty state, archive detail text, archive copy/share/delete buttons, and clear-all confirmation actions.

## Documentation

Documentation should remain concise:

- README and Chinese README mention `v0.11.0` as the current release and describe Local Session Archive briefly.
- DocC adds a `LocalSessionArchive` page and links it from Test Session Reports and the API overview.
- Roadmap records `v0.11 - Local Session Archive`.
- Changelog records `v0.11.0` only when the release is ready.
- Sample walkthrough adds archive verification steps.
- Privacy docs explicitly say archives persist locally until deleted and contain already-redacted report text.

Docs must clearly state that ConsoleDock does not save raw logs by default, does not upload archives, and does not guarantee crash-final logs.
