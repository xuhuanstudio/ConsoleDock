# ConsoleDock v0.4 Test Session Reports

Status: accepted design specification for the v0.4.0 implementation phase.

## Goal

ConsoleDock v0.4.0 should help a tester turn an on-device debugging session into a useful issue report without leaving the app or connecting Xcode.

The release builds on the v0.3 console and Debug Actions work. It does not change ConsoleDock into a remote testing platform, crash reporter, network inspector, or persistent log collector.

## Product Scope

v0.4.0 adds three connected capabilities:

- session metadata: local app and device context that explains where a log snapshot came from;
- manual markers: explicit timeline entries inserted by code or by a tester from the panel;
- issue report export: a user-initiated plain text report containing session metadata, diagnostics, markers, and redacted logs.

The feature remains local-only. ConsoleDock stores data in memory, redacts before storage, and only exports when the tester taps a share action.

## Non-Goals

This release must not add:

- default disk persistence;
- background upload;
- remote command delivery;
- automatic route discovery;
- parameterized Debug Actions;
- crash reporting;
- network request inspection;
- CocoaPods or XCFramework distribution;
- a public JSON Lines format commitment;
- complete Swift `Logger`, `os_log`, or Apple unified logging capture.

## Public API

### Swift

Add a session metadata model exposed through the Swift facade:

```swift
public struct SessionMetadata: Equatable {
    public let sessionIdentifier: String
    public let startedAt: Date?
    public let generatedAt: Date
    public let bundleIdentifier: String?
    public let appVersion: String?
    public let appBuild: String?
    public let processName: String
    public let operatingSystemVersion: String
    public let deviceModel: String
    public let localeIdentifier: String
    public let timeZoneIdentifier: String
}

public static var sessionMetadata: SessionMetadata { get }
```

Add a marker API:

```swift
public static func mark(_ message: String)
```

`mark(_:)` appends a native info-level entry with a stable marker prefix. It is safe when ConsoleDock is not running, following the existing native logging behavior.

No new public report formatter API is required in v0.4. Issue report formatting can stay internal until the format has real user feedback.

### Objective-C

Expose session metadata and markers through `ConsoleDockCore` / UIKit-compatible Objective-C APIs:

```objc
@interface CDKSessionMetadata : NSObject
@property (nonatomic, copy, readonly) NSString *sessionIdentifier;
@property (nonatomic, strong, readonly, nullable) NSDate *startedAt;
@property (nonatomic, strong, readonly) NSDate *generatedAt;
@property (nonatomic, copy, readonly, nullable) NSString *bundleIdentifier;
@property (nonatomic, copy, readonly, nullable) NSString *appVersion;
@property (nonatomic, copy, readonly, nullable) NSString *appBuild;
@property (nonatomic, copy, readonly) NSString *processName;
@property (nonatomic, copy, readonly) NSString *operatingSystemVersion;
@property (nonatomic, copy, readonly) NSString *deviceModel;
@property (nonatomic, copy, readonly) NSString *localeIdentifier;
@property (nonatomic, copy, readonly) NSString *timeZoneIdentifier;
@end

+ (CDKSessionMetadata *)sessionMetadata;
+ (void)mark:(NSString *)message;
```

The Objective-C names should keep the existing `CDK` prefix policy.

## Marker Semantics

A marker is just a normal stored log entry with a recognizable message prefix:

```text
[marker] Started checkout reproduction
```

Reasons for this design:

- markers automatically work with existing storage, redaction, truncation, search, filters, detail view, copy, and export;
- no migration is needed for the current `CDKLogEntry` model;
- Objective-C and Swift APIs stay small;
- the feature is easy to validate in current tests and sample UI smoke.

Marker messages still pass through default and app-specific redaction before storage. Empty or whitespace-only marker messages should normalize to a short default such as `[marker] Marker`.

## Session Metadata

Session metadata is generated locally from public platform APIs:

- `sessionIdentifier`: generated UUID for the current ConsoleDock runtime session;
- `startedAt`: set when ConsoleDock successfully starts and preserved until the next successful start session;
- `generatedAt`: report generation time;
- `bundleIdentifier`: `Bundle.main.bundleIdentifier`;
- `appVersion`: `CFBundleShortVersionString`;
- `appBuild`: `CFBundleVersion`;
- `processName`: `ProcessInfo.processInfo.processName`;
- `operatingSystemVersion`: `ProcessInfo.processInfo.operatingSystemVersionString`;
- `deviceModel`: `UIDevice.current.model` on UIKit platforms, otherwise `unknown`;
- `localeIdentifier`: `Locale.current.identifier`;
- `timeZoneIdentifier`: `TimeZone.current.identifier`.

The implementation should avoid private APIs and should not collect user identifiers, account identifiers, network addresses, raw environment variables, or file paths.

## Issue Report Format

The bundled UI adds `Share Issue Report` to the existing share action sheet.

The report is plain text and should include:

```text
ConsoleDock Issue Report
Generated: 2026-06-26T...

Session:
  Session ID: ...
  Started: ...
  Bundle ID: ...
  App Version: ...
  App Build: ...
  Process: ...
  OS: ...
  Device: ...
  Locale: ...
  Time Zone: ...

Diagnostics:
  Running: ...
  stdout: ...
  stderr: ...
  Floating Button: ...
  Release Builds: ...
  Limits: ...
  Redacted: ...
  Truncated: ...
  Partial: ...

Markers:
  [timestamp] [marker] ...

Logs:
  [timestamp] [native] [INFO] ...
```

Logs use the already-redacted stored entries. Markers should also appear in the full logs section because they are normal entries; the separate markers section is a convenience index.

`Share Visible Logs` and `Share All Logs` remain unchanged except for any shared internal formatter cleanup needed to avoid duplication.

## UIKit Behavior

The Logs page gains one additional navigation action:

- `Mark`: opens a small alert with a text field and actions `Cancel` and `Add Marker`;
- if the tester enters non-empty text, ConsoleDock appends a marker entry and refreshes the list;
- if the field is empty, ConsoleDock appends the default marker message;
- `Mark` is available while ConsoleDock is running.

The existing share button action sheet gains:

- `Share Visible Logs`;
- `Share All Logs`;
- `Share Issue Report`;
- `Cancel`.

The issue report shares all currently retained entries, not only visible entries. This avoids a tester accidentally sending a filtered report without surrounding context.

No new tab is required for v0.4. A separate Reports tab would overstate the feature and add navigation weight.

## Sample Apps

Swift and Objective-C sample apps should show the feature through small, realistic controls:

- add a sample marker button or Debug Action that writes a marker;
- include marker verification in the existing UI smoke flow;
- include issue report share action presence in UI smoke where practical without requiring an external share destination.

The samples should not simulate remote upload or issue tracker integration.

## Testing Plan

Unit tests:

- Swift `SessionMetadata` maps core metadata fields.
- Objective-C `CDKSessionMetadata` exposes expected fields.
- `sessionIdentifier` changes across fresh start sessions.
- `startedAt` is set after start and appears in issue reports.
- `mark(_:)` stores a native info entry with the marker prefix.
- empty marker messages use the default marker text.
- marker messages are redacted and truncated through the existing core path.
- issue report formatter includes session metadata, diagnostics, marker index, and logs.
- issue report formatter handles empty entries and missing bundle version fields.
- visible/all log snapshot behavior remains stable.

UI smoke:

- Swift sample can add a marker through the panel and see it in Logs.
- Objective-C sample can add or trigger a marker and see it in Logs.
- Share action sheet exposes `Share Issue Report`.
- Existing log detail, filters, pause/resume, clear, and Debug Actions flows still pass.

Release validation:

- `scripts/validate-release.sh` passes locally.
- GitHub `Release Validation` passes on the `v0.4.0` tag before a GitHub Release is created.
- Public release verification passes after release publication, allowing Swift Package Index access challenge if the external site returns one.

## Documentation Plan

Update only the documentation that users need for the new capability:

- `README.md`: short section for markers and issue reports;
- `README.zh-CN.md`: short synchronized summary;
- DocC: add or update a Test Session Reports page;
- `docs/sample-app-walkthrough.md`: manual verification steps;
- `docs/roadmap.md`: add v0.4 as Test Session Reports;
- `CHANGELOG.md`: add v0.4.0 entries before tagging.

Documentation must explicitly say:

- issue reports are generated locally;
- export is user-initiated;
- logs are already-redacted but not guaranteed safe;
- ConsoleDock does not upload reports or integrate with remote issue trackers.

## Implementation Shape

Prefer small additions over a broad rewrite:

- `ConsoleDockCore`: add `CDKSessionMetadata`, session start timestamp/identifier tracking, and `mark:`.
- `ConsoleDock`: map metadata into Swift, expose `mark(_:)`, and add an internal issue report formatter.
- UIKit: add Mark alert and issue report share option to `ConsoleDockLogsViewController`.
- Tests: extend existing package tests rather than creating large new test targets.
- Validators: update API surface validators for the new public Swift and Objective-C APIs.

Do not split UI files only for aesthetics. Split only if the v0.4 changes make a file harder to reason about.

## Acceptance Criteria

v0.4.0 is complete when:

- Swift and Objective-C callers can create markers.
- The bundled panel can create a manual marker.
- The bundled share sheet can create an issue report.
- The report includes session metadata, diagnostics, markers, and redacted logs.
- Samples and docs demonstrate the flow without implying remote upload or persistent storage.
- Unit tests and UI smoke cover the new behavior.
- Local release validation passes.
- Remote tag release validation passes.
- GitHub Release `v0.4.0` is published only after validation succeeds.
