# ConsoleDock 1.0 Readiness

ConsoleDock can reach `1.0.0` when the existing local debugging SDK is stable enough to support real Swift, Objective-C, and mixed UIKit app adoption without changing its product boundary.

This checklist is intentionally finite. It prevents the project from drifting into analytics, telemetry, remote commands, crash reporting, network inspection, CocoaPods, or XCFramework work before there is real demand.

## Stable Product Boundary

`1.0.0` should commit to these default behaviors:

- local-only, debug/test-first runtime behavior;
- Release builds disabled by default unless both compile-time and runtime opt-in gates are set;
- memory-first log storage, with only explicit bounded local archives and temporary support-report files;
- no automatic upload, telemetry, analytics, remote issue creation, or remote command delivery;
- no promise of complete Swift `Logger`, `os_log`, Apple unified logging, system log, or other-process log capture;
- Swift Package Manager as the supported public distribution channel.

## API Stability

Before tagging `1.0.0`, the public API should be reviewed as source-stable for normal app integrations:

- `ConsoleDock.Configuration`, lifecycle, diagnostics, entries, logging, markers, logger forwarders, issue reports, Support Reports, Local Session Archives, App Context, Debug Actions, and floating trigger controls;
- Objective-C-compatible `ConsoleDockCore` header APIs;
- Objective-C/UIKit facade APIs in `CDKConsoleDockUIKit` and related wrapper objects;
- API surface validators must pass and should only allow intentional public changes.

Any remaining breaking source cleanup should happen before `1.0.0`, not after it.

## Adoption Checks

The documented first-use path should remain short:

- Swift app: add the package, import `ConsoleDock`, call `ConsoleDock.start()`, and write one native log;
- Objective-C/UIKit app: add the package, call `CDKConsoleDockUIKit startWithConfiguration:error:`, and write one native log;
- existing logger: add one `ConsoleDock.LogForwarder` or `CDKLogForwarder` sink/appender at the logger boundary;
- feedback flow: generate a bounded Support Report on demand without showing the console panel to end users.

Docs should keep these paths easy to find in README, DocC, and migration guidance.

## Validation Gates

`1.0.0` should only be tagged after:

- `scripts/validate-release.sh` passes locally with a clean working tree;
- Swift and Objective-C sample UI smoke tests pass locally at least once for the final candidate;
- `main` branch CI passes on the exact release commit;
- tag Release Validation passes with `CONSOLEDOCK_RUN_UI_SMOKE=1`;
- post-release verification passes, allowing only documented Swift Package Index access-challenge warnings.

## Maintenance Shape

The codebase should remain reviewable:

- Swift facade tests stay split by product area rather than returning to one large file;
- release validation keeps the test-structure gate;
- new features after `1.0.0` are demand-driven and preserve the local-only default safety model;
- patch releases prefer small, verified fixes over moving tags.
